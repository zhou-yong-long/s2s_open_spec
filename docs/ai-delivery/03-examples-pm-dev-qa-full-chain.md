# PM / Dev / QA 全链路用例（叙事版）

以下用**同一虚构需求**串起来：`login-rate-limit`（登录接口限流）。假设业务仓库已完成 SDD 初始化，且启用了工作流类能力（如自检、评审、归档）。日期占位：`2026-05-11`。

本文**不写具体操作命令**；命令与自检方式见 [04-supplement-checklist.md](./04-supplement-checklist.md) 与 [KarmaStudio / SDD 集成交付说明](../karmastudio-sdd-delivery.md)。

---

## 一、需求是什么（一句话 + 边界）

**业务目标**：对登录接口做限流，降低撞库与暴力破解风险，同时不误伤正常用户（含峰值与弱网重试场景）。

**隐含约束**（示例，用于后续扯皮与验收）：限流维度（IP / 用户 / 设备）、阈值与窗口是否可配置、超限时的用户可见文案与错误码、与现有风控或网关策略的关系。

---

## 二、各角色产物、卡点与「下一阶段」

下表是**同一时刻的切片**：真实项目里各列会随时间推进；这里用「阶段 A → B → C」帮助对齐预期。

| 阶段 | 主要产物（Artifacts） | 当前常见卡点（示例） | 下一阶段（要推进到什么状态） |
|------|------------------------|----------------------|------------------------------|
| **PM** | 业务向规格：目标、范围、验收标准（AC）、上线/灰度与回滚叙述；状态从草稿到可评审 | AC 不可测或缺少边界；与法务/客服文案未对齐；与现有登录错误体验冲突未说明 | 规格进入「可评审」态，评审意见可收敛为变更清单 |
| **评审（跨角色）** | 评审记录、决议（通过 / 带条件通过 / 打回）；讨论线程或会议纪要 | PM 与工程对「限流语义」理解不一致；缺容量或压测承诺 | 对 PM 规格和/或工程规格给出明确决议，Dev 可开工 |
| **Dev** | 工程向规格：接口契约、组件/模块职责、与代码的关联（如 `linked_files`）；实现代码；必要时实现与规格对齐的变更说明 | 工程规格滞后于代码；契约与网关/客户端未对齐；环境开关与默认值未写清 | 工程规格审批通过，实现完成且关联文件可追溯，发布前对齐检查通过 |
| **QA** | 自功能规格派生的检查表、追溯矩阵（AC/边界 → 用例）、执行结果与签核 | `links.parent` 未指向权威功能规格；用例未覆盖「误伤」与观测；环境数据不可复现 | 追溯完整、执行完毕、签核结论明确，必要时反哺规格可测性 |
| **收尾** | 归档或完成态规格、发布说明中的可追溯条目 | 归档规则与「谁点完成」未约定 | 按团队流程进入完成/归档，规格与发布事实一致 |

## 三、模拟用户输入 → 期望输出（按角色）

语气模拟**真人在 IDE / 机器人里打字**，输出是**角色视角下应出现的结果类型**（具体文件名以团队模板为准）。

### 3.1 PM

| 用户输入（模拟） | 期望输出（类型） |
|------------------|------------------|
| 「帮我新建登录限流的业务规格，标题用英文 Login rate limit，今天要能进评审。」 | 新的 PM 向规格文件草案：含 Goals、AC 骨架、Rollout 段落占位；状态为草稿。 |
| 「AC 里补上：同一 IP 1 分钟超过 20 次返回 429，文案用我们统一错误格式。」 | 更新后的 AC 条目可测试、可自动化对接；与错误码/文案规范一致。 |
| 「我写得差不多了，按我们流程自检一下能不能标 ready。」 | 自检结果：要么列出缺口（例如缺灰度指标），要么规格可被标记为可评审（ready）。 |

### 3.2 Dev

| 用户输入（模拟） | 期望输出（类型） |
|------------------|------------------|
| 「PM 规格已定稿，帮我拉一份工程向的 login-rate-limit 规格，我要写接口和组件。」 | 新的工程向规格：Problem/Constraints 与 PM 结论对齐；Interfaces / Components 可填。 |
| 「限流做在网关层，登录服务只消费 header；帮我把 linked_files 列上我会改的 3 个文件。」 | 规格中关联路径/模块与真实改动计划一致，便于评审与后续 diff。 |
| 「实现完了，看看规格和代码有没有漂移，我准备提测。」 | 对齐报告或变更摘要：漂移项要么已用规格修订吸收，要么列为已知技术债并通知 QA。 |

### 3.3 QA

| 用户输入（模拟） | 期望输出（类型） |
|------------------|------------------|
| 「从当前功能规格生成 QA 检查表，父链接指向 specs/active/2026-05-11-login-rate-limit.md。」 | 新的 QA 文档：`links.parent` 正确；检查项从 AC 派生。 |
| 「把 Traceability 表里 AC-01、AC-02 映射到自动化用例 ID。」 | 表内可追溯单元格填满；未覆盖处标为「需数据 / 需环境」而非空白。 |
| 「预发跑完了，登录与注册回归通过，签 off。」 | Sign-off 表有结论、时间、环境；必要时对规格开「补充可测性」的备注而非口头传递。 |

## 四、哪些地方必须有人工介入

| 介入点 | 为什么机器/模板不够 |
|--------|---------------------|
| **业务取舍** | 限流阈值、误伤容忍度、与产品体验的权衡，需产品/业务负责人拍板。 |
| **评审决议** | 「带条件通过」的条件是否已落实，需人签字或等价流程。 |
| **环境与数据** | 预发/灰度账号、流量特征、监控看板是否就绪，常依赖运维或 QA 协调。 |
| **实现细节与线上事故责任** | 代码级错误处理、开关、回滚执行，需开发/值班人员承担。 |
| **签核与发布门禁** | QA sign-off、发布窗口、合规检查，通常绑定组织流程而非单一工具输出。 |

---

## 五、最小时间线（叙事，非命令）

1. PM 产出可评审业务规格 → 评审收敛。  
2. Dev 在决议基础上写工程规格并获批 → 实现与规格、代码关联一致。  
3. QA 派生检查表与追溯 → 执行 → 签核；缺口回到 PM/Dev 补齐。  
4. 按团队约定归档或标记完成，发布材料可引用同一批规格与签核。

## 六、与分支策略的关系（集成事实）

**sdd-cli 可执行文件**应来自分支 **`feature/sdd-karmastudio-pack`** 的构建产物；勿默认 **`main`** 已包含相同行为。业务仓库自身分支策略与 CLI 来源分支无关；详见 [01-prompt-install-sdd-cli.md](./01-prompt-install-sdd-cli.md)。
